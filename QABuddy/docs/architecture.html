Idempotent Incremental Ingestion

A manifest file (data/.ingest_manifest.json) stores SHA-256 content signatures for every file + source combination. On re-ingestion:

Load docs from source folder → Compute per-file SHA-256 → Compare vs manifest → NEW/CHANGED → embed + upsert → UNCHANGED → skip → REMOVED → delete from Qdrant

Stable UUID5 IDs derived from source_type + path + text ensure the same content always maps to the same Qdrant point — idempotent upserts with zero duplicates.

Embedding model: BAAI/bge-m3 produces both dense vectors (1024-dimension cosine similarity) and lexical sparse vectors (token-level BM25 weights) in a single encode pass. Both are stored as named vectors in Qdrant for hybrid retrieval.

Why Hybrid Search?

Dense search (semantic) excels at understanding meaning — "login button breaks" matches "authentication widget failure".

Sparse search (lexical/BM25) is essential for exact identifiers — VWO-2002, NullPointerException, #coupon-input. Without it, ticket IDs and exception names would be missed.

RRF merge combines both rankings without score calibration, producing the best of both worlds.

Why Cross-Encoder Rerank?

The vector search ranks by similarity, not relevance. A chunk that's similar to the query might not answer it.

The bge-reranker-v2-m3 cross-encoder reads the full (query, chunk) pair together, scoring actual relevance via sequence classification — not just vector proximity. This catches nuanced mismatches that embedding similarity misses.

Backend API Reference

GET /
Serves the chat application UI. Jinja2-rendered HTML with source list, mode selector, ingest panel, and chat thread.
GET /api/health
System health check. Returns LLM key status, model names, Qdrant mode (embedded/server), collection info, and chunk count.
GET /api/sources
File listing per source folder for the sidebar. Returns file list, file count, and detail counts for key sources: detail_count (e.g. 107 tests for Playwright, 500 test cases, 28 tickets) with detail_label.
GET /api/stats
Chunk counts per source type. Returns {total, by_source: {company_docs: 4, jira_tickets: 3, ...}, labels: {...}}
POST /api/chat
Main ask endpoint. Accepts {question, sources[], mode?, history[]}. Streams SSE events: meta → citations → token* → done | error. Each `[n]` citation maps to a source card with snippet and rerank score.
POST /api/search
Debug retrieval-only endpoint (no LLM). Returns embeddings, query rewrites, per-step timings, and ranked results with scores.
GET /api/ingest
Ingestion endpoint. Params: source, limit, force. Runs in a background thread, streams load → diff → embed → done progress events via SSE.

Key Design Decisions

1. BGE-M3: Why One Model for Both Dense + Sparse

Most hybrid RAG systems run two separate models: one for dense embeddings and one for sparse (BM25/Splade). BGE-M3 produces both in a single encode pass — half the latency, half the model memory. The dense vector handles semantic understanding while the sparse lexical weights give exact term matching (critical for ticket keys, exception names, test IDs like VWO-2002).

2. Qdrant Named Vectors: Zero-Custom-Index Hybrid

Qdrant's named vectors allow storing both dense and sparse embeddings in one point without maintaining separate collections. Queries hit both with one client call. The RRF fusion happens at the application layer for full control over the k parameter.

3. Reranker Bypass: Raw Transformers over FlagEmbedding

FlagEmbedding's FlagReranker API broke tokenizer compatibility in v1.4.0. The app bypasses it entirely — loading AutoTokenizer + AutoModelForSequenceClassification directly from HuggingFace transformers. More control, no dependency fragility.

4. Relevance Threshold Gate (0.22)

When the best reranker score falls below 0.22, the system responds with "That topic is not in my knowledge base" instead of generating from the LLM's parametric knowledge. This prevents hallucination and signals to users that the answer isn't grounded in their actual data.

5. Glossary Injection ≠ Embedding

Company-specific terms (VWO, RTM, SDET, POM) are defined in glossary.yaml and injected into the system prompt only. They are not embedded into Qdrant vectors. Why? These are domain translations, not retrievable knowledge. Embedding them would pollute the vector space with definitions that the LLM should simply know.

6. Idempotent Ingestion with SHA-256 Manifests

Re-ingestion diffs every file against a manifest of content signatures. Only changed files are re-embedded. UUID5 IDs ensure the same content → same Qdrant point → idempotent upserts with zero duplicates. This makes hourly auto-ingestion (Phase 2) practical.

7. SSE Streaming Everywhere

Both chat responses and ingestion progress are streamed via Server-Sent Events. This avoids HTTP timeouts on long-running operations (LLM generation can take 10-30s, ingestion can take minutes for large repos). The browser connects once and receives events as they happen.

8. Dual-Mode Qdrant: Zero Config Change

Set QDRANT_PATH → embedded mode (local dev, zero setup). Set QDRANT_URL → server mode (production, Docker Compose). The Store class handles both through the same QdrantClient API. No code changes needed to switch between local dev and production.
